# QA 审查意见：DEV-plan 前端开发任务拆分检查报告

**检查时间**：2026-04-08
**检查对象**：DEV-plan-PP-v1.0.md 第 8 节（前端开发任务拆分）
**参照标准**：原型（P00~P17）与 DESN-PP-v1.0.md 设计文档

整体来看，DEV-plan-PP-v1.0.md 中第 8 节的前端任务拆分与原型清单（P00-P17）完全对应，与后端的模块拆分也保持了高度的一致性。但深入检查前后端 API 交互、组件复用以及具体功能点后，发现以下几点遗漏和不一致之处：

### 1. 功能与 API 遗漏

1. **FE-FEED（校友圈前端）遗漏“删除动态”功能**
   - **问题说明**：后端 `feed` 模块明确提供了 `DELETE /feed/feed/{id}`（仅发布人可删除）接口。但在前端 `FE-FEED`（任务 FE-FEED-001 ~ FE-FEED-005）的子任务详情中，**完全没有**提及删除动态的 UI 交互和 API 对接。
   - **修改建议**：当前暂不实现发布人删除动态功能。

2. **FE-ADM（企业端商机管理）遗漏“下架商机”API 调用明确**
   - **问题说明**：`FE-ADM-006` 提到了 UI 上有“商机列表、上下架操作”，但在 `FE-ADM-007` (商机管理API对接) 的详情中，仅列出了 `GET /ent-admin/opportunities`、`PUT /ent-admin/opportunities/{id}` (编辑)、`POST /republish` 和 `DELETE`。
   - **修改建议**：在 `FE-ADM-006` 与 `FE-ADM-007` 中，将“删除操作”移除，统一替换为“下架/重新发布”操作，在 `FE-ADM-007` 详情中补充调用商机模块的下架接口 `POST /opp/opportunity/{id}/offline` 和重新发布接口`PUT /ent-admin/my-opportunities/{id}/republish`。

3. **FE-AUTH（认证模块）与 FE-PUB（公共模块）间的用户信息流转**
   - **问题说明**：`FE-PUB-001` 的公共 Header 需要根据用户角色显示“企业工作台”或“管理后台”的入口，同时也需要显示头像/用户名。但任务中没有明确在哪里统一调用 `GET /auth/me` 并将其存入全局状态（Pinia）。
   - **修改建议**：在 `FE-AUTH-008` (Token管理+路由守卫) 中明确补充：登录成功及应用初始化时调用 `GET /auth/me` 获取当前用户信息及权限，并存入 Pinia store。

### 2. 前端组件与技术细节缺失

1. **缺少“公共图片上传”组件任务**
   - **问题说明**：`FE-OPP-003` (发布商机) 和 `FE-FEED-003` (发布动态) 均涉及多张图片的上传功能（前端需要限制张数、预览、调用上传 API 拿 URL）。目前任务分散在各自模块中，容易导致重复造轮子。
   - **修改建议**：在 `FE-PUBLIC` 或项目初始化任务中抽取一个独立任务，开发一个通用的 `ImageUploader` 组件，统一处理图片压缩、上传和预览逻辑。

2. **消息通知（Badge）的实时刷新机制缺失**
   - **问题说明**：`FE-PUB-001` (公共 Header) 的通知铃铛红点只写了调用 API（`GET /msg/notifications/recent`）。如果在页面停留期间有新的商机联系、企业审核等通知，前端缺少更新机制。
   - **修改建议**：在 `FE-MSG` 或 `FE-PUB` 中补充说明通知更新机制，前端采用定时轮询机制，确保 Badge 数字的实时性。

3. **FE-SCH（搜索服务）的分页/响应结构描述存在歧义**
   - **问题说明**：后端 API 设计为 `GET /search?keyword=xxx&tab=opp|ent|feed`。但在后端 `SCH-002` 描述中返回结构是 `{opp: {...}, ent: {...}, feed: {...}}`。如果在 `FE-SCH-002` 搜索 API 对接时，每次切换 Tab 都需要分页加载，前端任务没有清晰说明如何结合 Tab 参数做单域的无限滚动或分页。
   - **修改建议**：明确 `FE-SCH-002` 中的数据加载逻辑，例如：首次搜索无 `tab` 参数时全量查询三个域的前置数据；切换 Tab 并向下滚动加载更多时，带上具体 `tab` 参数进行针对性的分页请求。

### 结论
任务划分在宏观页面层面（P00~P17）与原型和 DESN 完全对齐，没有页面级遗漏。建议根据上述意见（尤其是“校友圈删除缺失”和“公共图片上传组件”），修改 `DEV-plan-PP-v1.0.md` 后，即可作为 M3 阶段的前端开发实施依据。
