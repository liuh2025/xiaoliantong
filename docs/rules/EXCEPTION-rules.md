# 异常处理及需求变更规范（EXCEPTION-rules.md）

## 异常分级

| 级别 | 定义 | 处置方式 |
|------|------|---------|
| 阻断（BLOCK） | 违反红线，流程无法继续 | 立即停止，汇报 Orchestrator，修复后重新检查 |
| 警告（WARN） | 存在风险但不阻断流程 | 记录到 CHANGELOG，限期修复 |
| 提示（INFO） | 建议优化项 | 记录到 CHANGELOG，按优先级处理 |


---

## 需求变更流程

```
发现需求变更
    │
    ├── 评估影响范围（涉及哪些文档、代码、测试用例）
    │
    ├── 在 CHANGELOG 中记录变更内容和原因
    │
    ├── 更新受影响的文档（PRD/DESN/QA-test-plan）
    │   状态重置为"草稿"，重新走审批流程
    │
    ├── 评估是否需要回退已完成的开发工作
    │
    └── 审批通过后，更新 CLAUDE.md 当前阶段（如需回退阶段）
```


---

## 里程碑回退规则

| 场景               | 处置                         |
| ---------------- | -------------------------- |
| M3 开发中发现 M2 设计缺陷 | 回退到 M2，修正 DESN，重新审批，再推进 M3 |
| M4 测试中发现 M1 需求遗漏 | 回退到 M1，修正 PRD，走完整审批链路      |
| 紧急修复不影响架构        | 在当前阶段内处理，记录 CHANGELOG，无需回退 |

## Bug 修复流程

### 系统性调试四阶段

**Iron Law：找到根因前禁止提出修复方案。**

1. **根因调查：** 仔细读错误信息 → 稳定复现 → 检查近期变更 → 追踪数据流
2. **模式分析：** 找到可工作的类似代码 → 对比差异
3. **假设验证：** 提出单一假设 → 最小变更验证 → 确认后继续
4. **实现修复：** 先写失败测试 → 修复根因 → 验证

## 升级路径

| 问题类型              | 升级路径                                       |
| ----------------- | ------------------------------------------ |
| SubAgent 连续失败 3 次 | DEV-Implementer → TL → Orchestrator → User |
| 跨模块依赖冲突           | TL → Orchestrator → ARCH → User            |
| 需求理解偏差            | 任何角色 → Orchestrator → BA → User            |
| 红线违反              | 任何角色 → QA → Orchestrator → User            |
| 架构设计缺陷            | 任何角色 → ARCH → Orchestrator → User          |
